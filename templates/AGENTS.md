## 3. Architecture Rules

- Single Agent: Codex is the only execution agent. No Planner/Worker/Reviewer split.
- Single Model: One primary model for all reasoning and coding.
- Skills Instead of Agents: Capabilities are added as MCP Skills, not as sub-agents.
- Git First: Git is the long-term source of truth.
- Workspace First: All project state lives in .project/ directory. Chat history is not project state.
- Approval First: All high-risk operations require explicit human approval.

## 6. Testing and Verification

Before completing any code modification task, the following verification steps must be executed if applicable:

1. Lint — if a lint command is configured
2. Typecheck — if a typecheck command is configured
3. Unit tests — if test command is configured
4. Build — if a build command is configured

Verification results must be written to .project/reports/verification/.

## 7. Coding Standards

- Follow the existing code style and conventions in the repository.
- Match comment density, naming conventions, and code structure of surrounding files.
- Prefer small, focused modules with clear interfaces.
- Write tests for new functionality.
- Do not introduce unnecessary abstractions or patterns not already present.

---

## 8. Context Rules

- Always read the current task from .project/tasks/active/ before starting work.
- Always check git status before modifying files.
- Never overwrite user uncommitted changes without confirmation.
- Context Pack is generated by Harness OS before each run.
- Context sources are prioritized: AGENTS.md > .project/state/ > current task > related decisions.

---

## 9. State and Memory Rules

- Project state is stored in .project/state/.
- Task records are stored in .project/tasks/.
- Architecture decisions are stored in .project/decisions/ as ADR files.
- Run reports are stored in .project/reports/runs/.
- Session state is local and not committed to Git.
- Git is used for long-term project facts only.

---

## 10. Skill / Tool Rules

Only skills declared in the Context Pack may be used for task execution.

Available skills are managed by Harness OS and configured per project.

Skills are tools, not agents. Skills do not perform reasoning or planning.

---

## 11. Permission and Approval Rules

The following operations always require explicit human approval:

- Modifying AGENTS.md
- Modifying accepted Architecture Decision Records (ADRs)
- Deleting files or directories
- Adding major dependencies
- Running database migrations
- Deploying to any environment
- Pushing to the main/master branch
- Force pushing
- git reset --hard or git clean -fd
- Executing sudo commands
- Executing curl | sh or wget | sh
- Reading sensitive files (.env, credentials, private keys)
- Modifying security policy or governance rules

The following operations are allowed automatically:

- Reading workspace files
- Searching code
- Reading git status/diff/log
- Running declared test/lint/typecheck/build commands
- Writing to .project/reports/ and .project/context/

---

## 12. Git and Delivery Rules

- Always read git status before making changes.
- Never overwrite user uncommitted changes.
- Commit messages follow Conventional Commits format.
- Verification must pass before delivery.
- Run report must be generated before creating a PR.
- All commits, PRs, and deployments are governed by the approval rules above.

---

## 13. Security Rules

- Never read, print, or expose secrets (API keys, tokens, passwords, private keys).
- Never write .env values to reports, logs, context packs, or commit messages.
- Never execute unapproved network requests to unknown services.
- Never pipe downloaded scripts directly to shell.
- All file operations are restricted to the project workspace.

---

## 14. Task Completion Rules

A task is complete only when all of the following are satisfied:

1. The task's user instruction has been fulfilled.
2. All new code has been written and tested.
3. Verification (lint/typecheck/test/build) has passed or been accounted for.
4. Changed files have been recorded.
5. A task summary has been written.
6. A run report has been generated.
7. Risks and follow-up items have been documented.
